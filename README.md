# Java Profiling Demo

A small Spring Boot application that produces safe, controllable Java profiling scenarios for Splunk Observability Cloud demonstrations.

See [DEMO.md](DEMO.md).

The application provides:

- A lock-contention scenario with visibly blocked worker threads.
- A retained-memory scenario that behaves like a memory leak.
- A web interface and REST API for enabling and disabling each scenario.
- Automatic OpenTelemetry traces and JVM metrics through the Splunk Distribution of the OpenTelemetry Java agent.
- Splunk AlwaysOn CPU and memory profiling.
- A Splunk Distribution of the OpenTelemetry Collector service.
- Configurable safety limits for threads and retained memory.

## Requirements

The Docker workflow is the supported path and contains all Java, Maven, agent, and collector tooling:

- Docker Engine
- Docker Compose
- A Splunk Observability Cloud realm and ingest access token

For development without Docker, use Java 25 and Maven 3.6.3 or newer.

## Configure and run

Create your private environment file and edit the required values:

```shell
cp .env.example .env
```

At minimum, replace `SPLUNK_ACCESS_TOKEN` and set `SPLUNK_REALM` to your Observability Cloud realm. The `.env` file is gitignored; do not commit it.

Build and start the application and collector:

```shell
docker compose up --build
```

Open <http://localhost:8080>, then enable either scenario. Stop the deployment with `Ctrl+C`.

Both services use host networking because this environment's Docker bridge DNS cannot reach external resolvers. The application sends telemetry to the collector at `127.0.0.1:4317`, the collector uses the host resolver for Splunk endpoints, and the UI remains available on `DEMO_PORT` (8080 by default).

To run in the background:

```shell
docker compose up --build --detach
docker compose ps
```

Stop a background deployment:

```shell
docker compose down
```

Disabling a scenario returns it to normal. In particular, the memory scenario stops allocating and releases all references that it retained.

## Observability configuration

The app sends OTLP over gRPC to `127.0.0.1:4317` over the shared host network. The collector's maintained default configuration exports data to the Splunk realm selected in `.env`. Application logs are not exported because this demo is focused on traces, metrics, and profiling.

The main controls in `.env.example` are:

| Variable | Purpose |
|---|---|
| `SPLUNK_ACCESS_TOKEN` | Required Observability Cloud ingest token |
| `SPLUNK_REALM` | Required Observability Cloud realm, such as `us0` |
| `SPLUNK_API_URL` | Optional API endpoint override; normally derived from the realm |
| `SPLUNK_INGEST_URL` | Optional ingest endpoint override; normally derived from the realm |
| `SPLUNK_TRACE_URL` | Optional trace endpoint override; normally derived from the realm |
| `SPLUNK_HEC_URL` | Optional profiling/HEC endpoint override; normally derived from the realm |
| `OTEL_SERVICE_NAME` | APM service name |
| `OTEL_SERVICE_VERSION` | Resource service version |
| `OTEL_DEPLOYMENT_ENVIRONMENT` | APM deployment environment |
| `SPLUNK_PROFILER_ENABLED` | Enables AlwaysOn CPU profiling |
| `SPLUNK_PROFILER_MEMORY_ENABLED` | Enables AlwaysOn memory profiling |
| `SPLUNK_PROFILER_CALL_STACK_INTERVAL` | CPU stack sampling interval |
| `SPLUNK_PROFILER_MEMORY_EVENT_RATE` | Target memory allocation events per second |
| `OTEL_METRICS_EXPORTER` | Java agent metrics exporter; defaults to `otlp` |
| `OTEL_JAVAAGENT_DEBUG` | Enables verbose Java agent diagnostics |
| `SPLUNK_MEMORY_LIMIT_MIB` | Collector memory limit |

The Java agent and collector versions are pinned in `.env.example` for repeatable demos and can be changed there when upgrading.

## Verify

The Docker image build runs the automated test suite. After starting the deployment, confirm that both services are running and inspect their telemetry startup logs:

```shell
docker compose ps
docker compose logs otel-collector
docker compose logs profiling-demo
```

Check the running application and generate a trace:

```shell
curl --fail http://localhost:8080/actuator/health
curl --fail http://localhost:8080/actuator/health/liveness
curl --fail http://localhost:8080/actuator/health/readiness
curl --fail http://localhost:8080/api/scenarios
```

In Splunk Observability Cloud, locate service `profiling-demo` in APM, open a trace generated by one of these requests, and then open AlwaysOn Profiling. Profiles can take a short time to appear after a fresh startup.

## REST API

List scenarios and their live status:

```text
GET /api/scenarios
```

Enable a scenario:

```shell
curl --request PUT \
  --header "Content-Type: application/json" \
  --data '{"enabled":true}' \
  http://localhost:8080/api/scenarios/blocking-threads
```

Disable it:

```shell
curl --request PUT \
  --header "Content-Type: application/json" \
  --data '{"enabled":false}' \
  http://localhost:8080/api/scenarios/blocking-threads
```

The second scenario ID is `memory-leak`.

## Safety configuration

All workload controls have bounded validation in the application. Defaults can be overridden in `.env`:

| Variable | Default | Allowed range |
|---|---:|---:|
| `DEMO_BLOCKING_WORKER_COUNT` | `8` | 1–64 |
| `DEMO_BLOCKING_LOCK_HOLD_MILLIS` | `900` | 1–30,000 ms |
| `DEMO_BLOCKING_GAP_MILLIS` | `50` | 0–30,000 ms |
| `DEMO_MEMORY_ALLOCATION_MEGABYTES` | `1` | 1–16 MiB |
| `DEMO_MEMORY_ALLOCATION_INTERVAL_MILLIS` | `250` | 25–60,000 ms |
| `DEMO_MEMORY_MAX_RETAINED_MEGABYTES` | `128` | allocation size–512 MiB |
| `DEMO_PORT` | `8080` | Any available host port |

The Compose configuration caps the Java heap at 384 MiB by default, while the retained-memory scenario stops at 128 MiB. Keep the retained-memory limit comfortably below the configured heap.

## Demo reset

1. Disable both toggles in the web interface.
2. Confirm both cards show `Normal` and retained memory shows `0 MiB`.
3. If needed, restart from a clean process with `docker compose restart profiling-demo`.

No application state survives a restart.
